# 智摄 PRD V2.0

## 1. 项目基本信息

**项目名称：** 智摄  
**一句话定位：** 一个面向摄影 / 剪辑爱好者的个人私有化 AI 视频分析辅助系统。它不以云端重处理原视频为核心，而是通过**本地预处理 + 分析包生成 + ChatGPT / DeepSeek 手动深分析 + 项目内结果归档**，帮助用户看懂视频、拆解视频、复刻视频、优化视频。  
**产品形态：** Web 应用 + 本地预处理工具  
**部署方式：** 本土化轻量云服务器 + Docker Compose + 统一公网网关  
**维护方式：** 本地 Git 提交 / GitHub 推送后，服务器 `git pull + docker compose up -d --build` 更新部署  
**使用对象：** 当前以个人使用为核心，先做单用户版本  
**当前阶段：** 可直接用于后续技术方案、接口清单、数据库设计与开发任务拆分

---

## 2. 项目背景

最初设想是：用户将高画质高帧率视频上传到项目，由云服务器统一做视频预处理和 AI 分析。

但结合当前实际条件：
- 主要分析对象是**几十秒的高画质、高帧率视频**
- 当前云服务器配置仅为**2 核 2 GB**
- 项目当前仅服务个人使用
- 允许部分流程采用手动方式完成

因此，项目需要从一开始就避免走“云端重处理原视频”的重方案。

对当前场景而言，真正现实、稳定、可维护的路线是：

1. **原视频的重处理放在本地完成**，例如抽关键帧、镜头切分、字幕转写、缩略图生成。
2. **云端项目只承担轻量职责**，例如管理知识库、生成分析 prompt、接收分析结果、归档历史记录。
3. **深度视频理解主要借助大模型完成**，必要时由用户手动将分析包提交给 ChatGPT 或其他模型进行深分析。
4. **项目的价值重心不在“云端吃原视频”，而在“组织素材、规范分析流程、沉淀分析结果”。**

因此，智摄要解决的不是“构建一套重型视频分析云平台”，而是做一个适合个人创作工作流的：

**AI 审片、拆片、复刻与优化辅助系统。**

## 3. 项目目标

### 3.1 核心目标

智摄主要完成两件事：

#### 目标一：拆解参考视频
用户上传或导入一个喜欢的视频后，系统帮助输出：
- 视频整体风格标签
- 调色方向与参数倾向
- 剪辑思路与镜头节奏
- 视频大致片段结构
- 仿拍时应该准备哪些镜头
- 哪些片段更适合 AI 生成
- 一套适合交给 ChatGPT 深分析的结构化分析材料

#### 目标二：优化用户自己的视频
用户上传或导入自己剪好的视频后，系统帮助输出：
- 节奏是否拖沓
- 镜头是否重复或无重点
- 调色是否统一
- 曝光 / 白平衡 / 色彩倾向是否存在明显问题
- 开头、中段、结尾是否需要调整
- 补拍建议、删减建议、风格修正建议
- 一套可让 ChatGPT / DeepSeek 继续深挖的优化分析材料

### 3.2 本版目标重心

本版 PRD 的目标，不是做“自动全流程云端视频分析系统”，而是先做出下面这个**最稳闭环**：

**本地预处理 → 项目生成分析包 / Prompt → 用户手动交给 ChatGPT 深分析 → 将结果回填项目 → 项目完成结构化归档与复用**

### 3.3 产品价值

智摄的核心价值，不是替代所有人工操作，而是把原本混乱、零散、低复用的视频分析过程整理成统一流程：

- 把“我觉得这个视频很好看”变成“我知道它为什么好看”
- 把“我想模仿它但不会拆”变成“我知道该拍哪些镜头”
- 把“我觉得我剪得不对劲”变成“我知道问题在哪、怎么改”
- 把“每次都重新问 AI”变成“我有统一分析模板和历史记录”
- 把“临时灵感”变成“可积累、可复用的个人创作知识系统”

### 3.4 非目标

首版不做以下内容：
- 云端完整原视频重处理流水线
- 云端逐帧高精度分析引擎
- 完整视频编辑器
- 多人协作系统
- 社区、作品广场、点赞评论
- 复杂会员与支付系统
- 承诺 100% 准确反推原作者使用的具体滤镜 / LUT
- 承诺 100% 精准鉴别 AI 生成片段

### 3.5 关键产品约束

1. **原视频重处理默认发生在本地，不默认发生在云端。**
2. **云端项目默认只接收轻量分析包，不强依赖上传原视频。**
3. **ChatGPT 等模型在这里主要承担深分析角色，而非视频预处理角色。**
4. 系统输出应定义为：

**风格推测 + 原因说明 + 参数建议 + 可执行拆解 / 优化建议**

而不是伪装成绝对识别。

### 3.6 本土化生产部署目标

首版不为智摄单独重新设计一套公网发布体系，而是直接复用当前服务器已经验证过的上线思路：

- 本地项目目录按 `d:\apps\Photography` 维护
- 服务器运行目录统一收口到 `/opt/apps/photography`
- 智摄服务先通过 `127.0.0.1` 预览端口完成本机验证
- 验证通过后，再接入已有统一网关 `hiremate-caddy`
- 站点间通过 `shared_gateway` 互联，对外只暴露一个正式域名 `<PHOTOGRAPHY_DOMAIN>`
- 所有发布动作优先走 Git 更新与 Docker 重建，不直接在生产机临时改散文件

也就是说，智摄的产品设计需要从一开始就兼容“轻量服务器 + 统一网关 + 可回滚”的真实上线环境。

## 5. 核心使用场景

### 场景 1：拆解参考视频
用户本地选择一个喜欢的视频，通过本地预处理工具生成分析包，项目再输出：
- 风格标签
- 调色推测
- 镜头结构
- 拍摄建议
- ChatGPT 深分析 prompt
- 回填模板

### 场景 2：优化自己的视频
用户本地对自己的视频生成分析包，项目输出：
- 审片提示清单
- 节奏 / 镜头 / 调色 / 风格问题框架
- 可执行优化方向
- 供 ChatGPT 深分析的标准 prompt

### 场景 3：沉淀个人知识库
用户在系统中维护自己的剪辑知识库，例如：
- 旅行 vlog 常见镜头结构
- 城市夜景常见调色思路
- 人像视频肤色处理原则
- 适合 AI 补镜头的典型场景

系统在生成 prompt 和回填结果整理时，可以自动引用知识库内容。

### 场景 4：手动 AI 深分析
系统生成分析包与标准化 prompt 后，用户手动把以下内容交给 ChatGPT：
- 关键帧 / Contact Sheet
- Transcript / 字幕文本
- 分镜摘要
- Prompt 模板

ChatGPT 返回结果后，用户将结果回填进项目，项目完成归档、整理、搜索和后续复用。

## 6. 产品定位重定义

智摄不是“云端视频处理平台”，而是：

**一个围绕个人视频创作工作流设计的 AI 分析中台。**

它的核心职责是：
- 帮用户把视频拆成适合 AI 理解的材料
- 帮用户生成高质量分析 prompt
- 帮用户承接 ChatGPT / DeepSeek 的分析结果
- 帮用户把这些结果标准化、归档、沉淀为知识

换句话说，智摄更像：

**本地预处理器 + Prompt 组织器 + 分析结果归档器 + 知识库辅助器**

而不是一个重型视频云计算系统。

## 8. 系统架构原则

### 8.1 总体架构

智摄采用“两段式架构”：

#### 第一段：本地预处理
由本地工具负责：
- 读取原视频
- 提取元信息
- 抽关键帧
- 做镜头切分
- 生成 Contact Sheet
- 提取音频
- 可选转写字幕 / transcript
- 产出分析包

#### 第二段：云端轻服务
由云端项目负责：
- 接收分析包
- 检索知识库
- 生成分析 prompt
- 保存分析任务
- 接收用户回填的模型分析结果
- 展示报告
- 保存历史记录

#### 第三段：外部大模型深分析
由用户手动完成：
- 将分析包内容和 prompt 交给 ChatGPT / DeepSeek
- 获取模型输出
- 将结果粘贴 / 上传回项目归档

### 8.2 架构设计原则

1. **重处理留在本地，轻分析留在云端**
2. **云端不依赖高配置服务器**
3. **首版优先保证工作流顺畅，而不是自动化极限**
4. **所有分析结果尽量结构化，便于归档和复用**
5. **产品要能在 2 核 2 GB 服务器上稳定运行**

### 8.3 生产部署原则

1. **公网入口统一收口**：智摄不直接占用公网 `80/443`，正式对外入口统一由现有 Caddy 网关承接。
2. **先预览后切流**：每次上线先在服务器本机端口验证，再补网关路由，不把未验证版本直接暴露公网。
3. **服务目录固定**：代码、环境变量、存储、日志都围绕 `/opt/apps/photography` 组织，避免散落部署。
4. **更新路径固定**：默认走 Git 拉取 + Docker 重建，避免手工替换文件导致环境漂移。
5. **回滚路径清晰**：任何涉及网关、数据库或存储结构的升级，都必须允许快速回退到上一版目录或备份。
6. **本土化环境优先**：域名、HTTPS、备案、时区、日志与路径命名都以国内云服务器长期运行为前提设计。

## 9. 详细功能需求

## 9.1 本地视频预处理工具

### 功能描述
本地工具负责读取原视频，并生成适合大模型分析的轻量分析包。

### 输入
- 原视频文件（优先支持 MP4、MOV）
- 可配置分析模式：拆解参考视频 / 优化我的视频
- 可配置抽帧密度与镜头切分敏感度

### 处理逻辑
本地工具自动完成：
- 提取视频元信息：时长、分辨率、帧率、码率
- 按时间间隔抽帧
- 做镜头切分 / 片段分段
- 每段选取代表帧
- 生成关键帧缩略图
- 生成 Contact Sheet
- 可选提取音频
- 可选转写字幕 / transcript
- 压缩输出资源，降低上传负担

### 输出
建议输出一个标准分析包目录，例如：
- `metadata.json`
- `segments.json`
- `transcript.txt`
- `frames/`
- `contact_sheet.jpg`
- `task_manifest.json`

### 价值
- 避免高码率原视频直接上传云端
- 降低服务器负载
- 让 ChatGPT 更容易理解视频结构
- 提高后续分析 prompt 的质量

---

## 9.2 分析包生成模块

### 功能描述
系统围绕本地预处理结果，生成标准化“分析包”，作为后续 AI 深分析的核心输入。

### 分析包包含内容
- 视频元信息
- 关键帧图片
- 镜头分段结果
- 字幕 / transcript
- 分析模式标记
- 分析目标说明

### 设计要求
- 结构稳定
- 文件体积可控
- 命名统一
- 便于上传、打包、导出和复用

### 验收标准
用户完成本地预处理后，能够一键得到一套清晰、可直接交给 AI 的分析包。

---

## 9.3 Prompt 生成模块

### 功能描述
系统根据任务类型、知识库内容和分析包摘要，自动生成适合提交给 ChatGPT / DeepSeek 的 prompt。

### 目标
让用户无需每次手动组织复杂提示词，直接复制即可使用。

### Prompt 类型
#### 拆解模式 Prompt
用于参考视频分析，要求模型输出：
- 视频风格标签
- 调色推测
- 剪辑结构
- 片段拆解
- AI 生成片段提示
- 仿拍建议

#### 优化模式 Prompt
用于用户自己视频分析，要求模型输出：
- 总体判断
- 必须改
- 建议改
- 可选优化
- 调色建议
- 剪辑建议
- 风格提升建议

### 输出形式
系统应支持导出：
- 一段完整 prompt 文本
- 建议附带的图片 / 文本清单
- 建议追加追问列表
- 回填结果模板

---

## 9.4 拆解模式（参考视频分析）

### 功能目标
帮助用户拆解一个参考视频的风格、调色、结构与复刻路径。

### 需要组织给模型的分析维度
- 风格标签
- 调色方向
- 画面质感倾向
- 剪辑节奏
- 镜头时长与节奏变化
- 分镜结构
- 情绪推进方式
- AI 生成片段可能性
- 仿拍方案

### 期望输出结构
1. 总体结论
2. 风格标签
3. 调色推测（风格 + 原因 + 参数）
4. 剪辑结构分析
5. 片段拆解
6. 疑似 AI 生成片段提示
7. 仿拍 / 复刻方案

### 关键要求
调色与风格相关输出必须尽量采用：

**可能风格 + 判断原因 + 建议参数 + 可选 LUT / 调色方向**

而不是只输出模糊标签。

---

## 9.5 优化模式（我的视频分析）

### 功能目标
帮助用户从审片角度分析自己的视频问题，并得到可执行的修改建议。

### 分析维度
#### 结构问题
- 开头是否弱
- 中段是否拖沓
- 结尾是否没收住
- 情绪推进是否不足

#### 镜头问题
- 镜头是否重复
- 主体是否不清晰
- 景别切换是否单一
- 运动镜头是否不稳定

#### 剪辑问题
- 节奏是否过快 / 过慢
- 转场是否突兀
- 卡点是否准确
- 是否存在无意义停顿

#### 调色问题
- 色彩是否统一
- 白平衡是否稳定
- 对比度是否过硬或过平
- 高光是否溢出
- 肤色是否自然
- 暗部是否发灰

#### 风格问题
- 想表达的情绪和实际呈现是否一致
- 题材和配色是否匹配
- 画面高级感是否不足
- 参考风格模仿是否到位

### 期望输出结构
建议分成三层：
- **必须改**：影响观看体验的问题
- **建议改**：明显提升质感的问题
- **可选优化**：进一步拔高风格的问题

同时每条建议都要包含：
- 问题描述
- 原因判断
- 修改建议
- 修改后的预期效果

---

## 9.6 知识库模块

### 功能目标
为 Prompt 生成和结果整理提供稳定的创作知识支撑。

### 知识库内容建议
#### 调色知识
- 不同题材适合的色彩倾向
- 常见滤镜风格说明
- 不同时间 / 天气 / 场景如何调色
- 曝光、对比、高光、阴影、饱和度经验规则

#### 剪辑知识
- Vlog、旅行、人像、城市感、情绪片的节奏特点
- 常见开头 / 中段 / 结尾结构
- 转场使用场景
- 卡点和非卡点剪法区别

#### 镜头知识
- 空镜、特写、中景、运动镜头用途
- 哪些题材需要环境镜头
- 哪些内容适合慢动作或快速切换

#### AI 生成知识
- 什么样的画面适合 AI 生成
- 什么样的画面更适合实拍
- AI 补镜头典型场景
- AI 生成画面与实拍混剪注意事项

### 维护方式建议
当前阶段推荐：
- 知识库存放在仓库内
- 采用 Markdown / JSON / YAML 管理
- Git 提交即可更新

## 9.7 手动 ChatGPT 深分析工作流

这是 V2 中新增的核心模块。

### 模块目标
为用户提供一套规范、稳定、低门槛的“手动 AI 深分析”流程。

### 标准流程
1. 用户用本地工具对视频做预处理
2. 项目接收或读取分析包
3. 项目自动生成：
   - 分析摘要
   - Prompt
   - 推荐附带图片清单
   - 推荐追问问题
   - 回填模板
4. 用户将这些内容手动交给 ChatGPT / DeepSeek
5. 获得模型输出后，将结果回填项目
6. 项目将结果结构化保存并展示

### 项目需要提供的支持能力
- “复制 Prompt”按钮
- “导出分析包说明”按钮
- “推荐上传哪些图”提示
- “推荐追问”列表
- “粘贴分析结果”输入区
- 回填后自动生成标准报告

### 价值
- 非常适合当前 2 核 2 GB 服务器条件
- 避免产品对不稳定的视频直传能力产生依赖
- 大幅降低开发和运维复杂度
- 能快速验证产品价值

## 14. 非功能需求

### 14.1 服务器配置适配目标
本项目首版应明确以**2 核 2 GB 云服务器可稳定运行**为约束条件。

### 14.2 性能要求
- 云端不承担原视频重处理
- 云端主要处理文本、图片、JSON 和报告展示
- 任务创建、Prompt 生成、回填归档应足够流畅
- 单用户场景优先，不考虑高并发

### 14.3 安全性
- 系统仅个人使用，也应有基础登录保护
- 不在前端暴露敏感配置
- 分析包上传做基本类型校验
- 历史记录和知识库支持本地备份

### 14.4 可维护性
- 所有配置环境变量化
- Docker 一键启动
- Git 更新后快速重启部署
- 知识库与代码分层管理
- Prompt 模板独立可维护

### 14.5 生产部署与回滚要求
- 服务器部署目录固定为 `/opt/apps/photography`
- 统一公网网关事实来源固定为 `/opt/apps/hiremate/Caddyfile`
- 智摄服务默认只暴露本机预览端口，推荐先使用 `127.0.0.1:18120`
- 正式公网入口统一为 `https://<PHOTOGRAPHY_DOMAIN>`，页面与 API 共用同一域名
- 重要发布前需备份 `.env`、`docker-compose.yml`、SQLite 文件与存储目录索引
- 不使用 `docker compose down -v` 这类会连带删除卷数据的危险动作

### 14.6 本土化环境要求
- 服务器时区统一为 `Asia/Shanghai`
- 所有时间展示、日志与导出默认按中国时区解释
- 正式公网域名应优先满足国内备案与 HTTPS 可签发条件
- 首版默认兼容腾讯云 / 阿里云 / 华为云等国内常见 Linux 云服务器环境

## 15. 技术方向建议（非详细技术方案）

### 15.1 总体策略
考虑到当前服务器只有 2 核 2 GB，首版不建议采用重架构，而应采用：

**本地预处理工具 + 云端轻量 Web 服务**

### 15.2 云端推荐技术栈
- 后端：FastAPI
- 页面：Jinja2 模板或轻前端
- 数据库：SQLite
- 文件存储：宿主机挂载目录（建议映射到 `/opt/apps/photography/volumes/storage`）
- 知识库：Markdown / JSON / YAML + 简单检索
- 部署：Docker Compose + `shared_gateway` 共享网络
- 公网入口：复用现有 Caddy 统一网关
- 反向代理：项目自身不必再单独承担公网 HTTPS，优先接入已有网关

### 15.3 本地工具建议
- FFmpeg
- PySceneDetect 或类似镜头切分能力
- 可选字幕转写模块
- 分析包生成脚本

### 15.4 首版不建议引入
- PostgreSQL
- Redis
- Celery
- 独立向量数据库
- MinIO
- 重型前后端分离体系

## 16. 风险与限制

### 风险 1：用户误以为项目会自动分析原视频
**解决方式：** 在产品说明中明确：首版核心是“本地预处理 + 手动 AI 深分析”。

### 风险 2：用户把 ChatGPT 当成视频预处理器
**解决方式：** 明确 ChatGPT 在工作流中的角色是“深分析器”，不是“预处理器”。

### 风险 3：手动流程可能增加操作成本
**解决方式：** 用高质量 Prompt、标准模板和一键复制功能，把手动操作压到最低。

### 风险 4：分析结果质量受 Prompt 和分析包质量影响
**解决方式：** 固定分析包结构、固定 Prompt 模板、逐步沉淀知识库。

### 风险 5：个人项目容易越做越大
**解决方式：** 首版只聚焦三个核心能力：生成分析包、生成 Prompt、归档结果。

## 19. 成功标准

1. 本地视频可以稳定生成分析包。

6. 整个流程在 2 核 2 GB 服务器条件下仍能稳定运行。
7. 服务器能通过 `git pull + docker compose up -d --build` 完成稳定更新。

## 20. 最终产品定义

智摄不是一个重型视频分析平台，而是：

**一个适配个人设备与轻量服务器条件的 AI 视频分析工作流系统。**

它的核心竞争力不在“云端吃掉原视频”，而在于：
- 可理解
- 可执行
- 可复刻
- 可归档
- 可复用

**首版只做三件事：本地预处理、生成 Prompt、回填归档，其余能力都围绕这三件事服务。**
