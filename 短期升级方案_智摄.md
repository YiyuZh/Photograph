# 智摄短期升级方案

## 1. 文档定位

本方案用于在 **智摄首版 MVP 已完成、正在进入真实部署与使用阶段** 的前提下，规划接下来 **4 到 6 周** 的短期升级方向。

它基于以下文档与现状整理：

- `智摄_prd_v_1.md`
- `首版开发任务拆分清单_智摄.md`
- `云端轻服务架构设计文档_智摄.md`
- 当前已经完成的首版代码、部署手册、任务记录与上线实践

这份方案的目标不是推翻首版重来，而是：

1. 稳住已经跑通的主链路
2. 补齐 PRD 中最影响体验与可维护性的短板
3. 让项目更适合真实长期使用与迭代

---

## 2. 当前基线

结合当前项目状态，智摄已经具备以下基础能力：

- FastAPI + Jinja2 + SQLite 的单体轻服务架构已落地
- 单用户登录保护已完成
- 任务创建、分析包上传解析、Prompt 生成、结果回填、报告生成、历史查看已打通
- 知识库、模板、设置、健康检查、Docker Compose、部署手册已具备基础版本
- 项目已进入真实云服务器部署与正式域名接入阶段
- 本地预处理 + 云端轻服务 + 手动 AI 深分析 + 回填归档 这条核心链路已经成立

当前仍然存在的短期问题：

- 上线流程虽然已基本跑通，但构建依赖、安装速度、网关接入、容器网络互通还需要继续稳固
- 分析包导入、Prompt 生成、报告输出虽然可用，但在“解释性、容错性、可复用性”上还偏首版
- 知识库与模板能力仍偏静态，距离真正形成个人复用中台还有空间
- 本地预处理工具与云端分析包协议虽已成型，但还缺更强的版本契约和兼容性校验
- 页面体验已完成两轮精修，但仍有一部分状态页、错误态和细节流程没有彻底标准化

---

## 3. 短期升级目标

短期阶段的核心目标聚焦 4 件事：

### 目标 1：把首版从“能跑”提升到“稳定可上线”

重点解决：

- 构建依赖解析稳定性
- 国内服务器安装速度
- Caddy 接入、DNS、`APP_BASE_URL`、容器网络的联动一致性
- 部署、回滚、冒烟测试的执行确定性

### 目标 2：把主链路从“打通”提升到“更顺手”

重点解决：

- 分析包导入时的缺失提示与错误定位
- Prompt 版本切换与复用体验
- 结果回填与报告阅读体验
- 任务状态可见性和导向性

### 目标 3：把知识库与模板从“可用”提升到“可沉淀”

重点解决：

- 知识条目分类、标签、搜索、引用关系
- Prompt 模板默认绑定与任务模式联动
- 个人经验沉淀的可维护性

### 目标 4：把本地工具与云端服务从“配合使用”提升到“协议稳定”

重点解决：

- 分析包结构版本化
- 本地工具输出与云端解析的兼容性校验
- 示例夹具与验收样本标准化

---

## 4. 升级原则

短期升级必须继续遵守 PRD 的原始约束：

1. 不做重型云视频处理平台
2. 不引入 Redis、Celery、PostgreSQL、MinIO、向量数据库等重组件
3. 继续坚持“本地重处理，云端轻服务”
4. 继续保持单用户、低配置服务器可运行
5. 继续复用统一公网网关，不单独新造发布体系
6. 优先提升稳定性、可读性、可维护性，而不是炫技

---

## 5. 短期升级排期

## 阶段 A：生产稳定化

建议周期：第 1 到第 2 周

### 目标

先把“真实上线部署”相关的不确定性压下去，避免后面每次更新都重新踩坑。

### 重点事项

- 收口 Python 依赖版本，避免 `FastAPI / Pydantic / pydantic-settings` 组合再次冲突
- 增加可切换的国内 PyPI 镜像策略，至少支持文档级与 Docker 构建级两种方案
- 完善部署手册中的正式域名、Caddy 站点块、`.env`、容器网络、回滚路径说明
- 补一份更执行化的冒烟测试步骤，覆盖：
  - 登录
  - 新建任务
  - 上传示例分析包
  - 生成 Prompt
  - 回填结果
  - 生成报告
  - 正式域名访问验证
- 增加服务器侧最小排障脚本或命令清单，例如：
  - `docker compose ps`
  - `docker compose logs`
  - `curl /api/health`
  - `git ls-remote`

### 交付物

- 更新后的 `requirements.txt`
- 更新后的部署手册
- 更明确的冒烟测试清单
- 可复用的构建与排障命令模板

### 验收标准

- 同一台低配服务器可以稳定完成一次从 `git pull` 到 `docker compose up -d --build` 的升级
- 正式域名与本机预览都能被清晰验证
- 遇到依赖或网关问题时，有固定排查路径

---

## 阶段 B：主链路体验增强

建议周期：第 2 到第 3 周

### 目标

把“上传分析包 -> 生成 Prompt -> 回填结果 -> 生成报告”这条链路做得更顺手。

### 重点事项

- 优化分析包导入反馈：
  - 哪些文件存在
  - 哪些文件缺失但可降级继续
  - 哪些文件缺失会阻断导入
- 在任务详情页增强分析包摘要展示：
  - metadata 摘要
  - segments 统计
  - 关键帧 / Contact Sheet 展示
  - transcript 可用性提示
- 增强 Prompt 工作台：
  - 当前版本与历史版本对比
  - 模板来源说明
  - 推荐追问可复制
  - 推荐附带素材列表更明确
- 增强结果回填：
  - 区分粘贴文本与上传文件来源
  - 保留原始结果与整理结果的对应关系
- 增强报告输出：
  - 总结段更稳定
  - 拆解 / 优化结构更清晰
  - 导出 Markdown / TXT 更统一

### 交付物

- 任务详情增强版
- Prompt 页面增强版
- 回填与报告展示增强版

### 验收标准

- 用户能明显看懂任务当前走到哪一步
- 用户不需要翻多个页面也能理解分析包是否可用
- Prompt、结果、报告三者之间的关系更清晰

---

## 阶段 C：知识库与模板增强

建议周期：第 3 到第 4 周

### 目标

把知识库与模板从“静态辅助”升级成“真正能复用的个人创作中台”。

### 重点事项

- 知识库条目补充更清晰的：
  - 分类
  - 标签
  - 适用场景
  - 关联任务类型
- 优化知识库搜索与筛选：
  - 分类筛选
  - 关键词搜索
  - 相关条目推荐
- 模板能力增强：
  - `reference` / `improve` 模式默认模板绑定
  - 模板版本管理
  - 模板说明与适用条件
- 在 Prompt 生成结果中明确显示：
  - 引用了哪些知识条目
  - 使用了哪个模板版本

### 交付物

- 知识库增强版数据结构与页面
- 模板管理增强版
- Prompt 生成链路中的知识引用展示

### 验收标准

- 用户能感知到“这次 Prompt 为什么这么生成”
- 不同任务模式下模板与知识注入更稳定
- 个人知识资产开始具备真正复用价值

---

## 阶段 D：本地工具契约与测试闭环

建议周期：第 4 到第 6 周

### 目标

让本地预处理工具与云端轻服务之间形成更稳定的协作契约。

### 重点事项

- 为分析包补 `schema_version` 或等价版本字段
- 明确分析包最小必需文件、可选文件、降级策略
- 增加分析包校验逻辑与报错文案
- 补充标准测试夹具：
  - 最小可导入包
  - 缺 transcript 包
  - 缺 segments 包
  - 结构错误包
- 为本地工具与云端解析增加兼容性测试说明
- 在文档中明确“本地工具升级时，云端需同步检查哪些字段”

### 交付物

- 分析包协议版本说明
- 测试夹具增强版
- 云端兼容性校验说明

### 验收标准

- 本地工具升级不会轻易把云端导入逻辑打崩
- 出现不兼容时，系统能尽早报错，而不是静默失败
- 新任务可以复用标准测试包快速验收

---

## 6. 明确不纳入短期升级的内容

以下内容不建议在短期阶段展开：

- 云端原视频重处理
- 自动调用大模型 API 作为主流程
- 多用户协作
- 支付、会员、社区功能
- 复杂搜索引擎或向量检索体系
- 前后端彻底分离重构

这些方向会明显抬高维护成本，也偏离当前 PRD 的轻量路线。

---

## 7. 建议的短期优先级排序

如果资源有限，推荐按下面顺序推进：

1. 阶段 A：生产稳定化
2. 阶段 B：主链路体验增强
3. 阶段 C：知识库与模板增强
4. 阶段 D：本地工具契约与测试闭环

原因很简单：

- 先稳住上线，才值得继续打磨体验
- 先打磨主链路，知识库沉淀才有意义
- 先补协议契约，后续迭代才不容易反复返工

---

## 8. 阶段完成后的目标状态

如果这轮短期升级完成，智摄应该达到的状态是：

- 可以在当前低配云服务器环境中更稳定地部署与更新
- 真实用户使用主链路时明显更顺畅
- Prompt、知识库、模板、报告之间的关系更清晰
- 本地工具与云端之间的协作协议更稳定
- 项目从“首版可运行”进入“可持续迭代”的阶段
