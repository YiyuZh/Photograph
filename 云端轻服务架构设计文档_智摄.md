# 云端轻服务架构设计文档

## 1. 文档定位

本文档用于定义 **智摄项目中的云端轻服务架构方案**。

该架构的目标不是构建一个重型视频分析平台，而是在当前明确的资源约束下，为整个智摄工作流提供一个稳定、轻量、可维护的云端中台。

当前前提已经明确：
- 云服务器配置为 **2 核 2 GB**
- 项目当前以 **单用户个人使用** 为核心
- 原视频重处理放在本地完成
- 云端不负责高码率视频抽帧、镜头切分、转写等重任务
- 深分析由用户手动借助 ChatGPT / DeepSeek 完成
- 生产环境沿用既有国内云服务器部署口径：`/opt/apps/<project>` + `shared_gateway` + 统一 Caddy 公网入口
- 智摄服务未来部署目录固定为 `/opt/apps/photography`
- 公网网关事实来源固定为 `/opt/apps/hiremate/Caddyfile`

因此，云端轻服务的职责应聚焦在：

- 接收与管理分析包
- 检索和组织知识库内容
- 生成标准化 Prompt
- 承接用户回填的大模型分析结果
- 生成和展示结构化报告
- 保存历史任务、模板和配置

它本质上是一个：

**AI 视频分析工作流管理中台**

而不是一个：

**重型视频处理计算平台**

---

## 2. 架构设计目标

### 2.1 核心目标

云端轻服务需要完成以下目标：

1. 在低配置服务器上稳定运行
2. 接收本地预处理工具输出的标准分析包
3. 基于任务类型和知识库生成高质量 Prompt
4. 支持用户将 ChatGPT / DeepSeek 的分析结果回填系统
5. 将原始结果整理为结构化报告
6. 保存历史任务，便于后续搜索和复用
7. 通过 Docker 部署并支持 Git 更新维护

### 2.2 设计原则

1. **轻量优先**：优先确保单用户稳定可用，不上不必要的重组件。
2. **文本与图片为核心**：云端主要处理 JSON、文本、图片和报告，不处理原始视频重任务。
3. **模块解耦**：分析包、本地工具、Prompt 生成、结果回填彼此通过清晰的数据结构连接。
4. **低维护成本**：适配个人部署，配置简单，启动简单，更新简单。
5. **可扩展**：在未来服务器升级后，允许逐步接入更多自动化能力。
6. **以工作流为中心**：系统优先优化“从分析包到报告”的链路，而不是单点炫技功能。
7. **复用既有上线体系**：不为智摄单独再造一套公网入口、切流方式和发布流程。

## 3. 架构边界

## 3.1 云端轻服务负责的内容

云端轻服务负责：

- 用户登录保护
- 新建分析任务
- 接收分析包
- 读取分析包元信息
- 保存关键帧、Contact Sheet、segments、transcript 等轻量文件
- 基于模板与知识库生成 Prompt
- 展示 Prompt 页面与复制按钮
- 接收分析结果回填
- 将结果结构化整理为报告
- 保存历史任务、Prompt、回填记录、报告
- 管理知识库与模板

## 3.2 云端轻服务不负责的内容

云端轻服务不负责：

- 原视频重上传与重存储
- 云端抽帧
- 云端镜头切分
- 云端音频转写
- 云端逐帧分析
- 本地大模型推理
- 多用户协作体系
- 高并发任务调度
- 重型向量数据库与复杂检索系统

也就是说，云端的角色是：

**组织、管理、生成、归档**

而不是：

**重算、重处理、重推理**

---

## 4. 整体架构总览

智摄的整体系统应分为三层：

### 第一层：本地预处理层
由本地工具负责：
- 读取原视频
- 抽帧
- 镜头切分
- 生成 Contact Sheet
- 提取音频 / transcript
- 输出分析包

### 第二层：云端轻服务层
由云端项目负责：
- 接收分析包
- 管理任务与数据
- 检索知识库
- 生成 Prompt
- 展示回填入口
- 保存结果与报告

### 第三层：外部大模型分析层
由用户手动操作：
- 将分析包和 Prompt 交给 ChatGPT / DeepSeek
- 获取分析输出
- 回填至云端项目

### 总体关系

```text
本地视频
  ↓
本地预处理工具
  ↓
分析包
  ↓
云端轻服务
  ↓
生成 Prompt / 模板 / 追问建议
  ↓
用户手动提交到 ChatGPT / DeepSeek
  ↓
模型输出结果
  ↓
回填云端
  ↓
结构化报告 / 历史归档 / 知识沉淀
```

1. 认证与基础访问控制模块

4. Prompt 生成模块
5. 知识库模块
6. 回填结果接收模块
7. 报告生成与展示模块

## 6. 模块详细设计

## 6.1 认证与基础访问控制模块

### 功能
为个人使用场景提供基础登录保护，避免系统裸露在公网。

### 首版目标
- 单用户登录
- 受保护页面访问控制
- 基础会话管理

### 建议方式
- 用户名 + 密码登录
- 服务端 Session 或简化 Token 机制
- Cookie 管理登录状态

### 首版要求
- 页面访问必须受保护
- 未登录时不能查看历史记录、知识库和报告
- 不追求复杂 RBAC 权限系统

---

## 6.2 任务管理模块

### 功能
管理每一次“拆解分析”或“优化分析”任务。

### 任务类型
- `reference`：拆解参考视频
- `improve`：优化我的视频

### 核心能力
- 创建任务
- 读取任务详情
- 修改任务状态
- 删除任务
- 按时间查看任务列表
- 查看任务绑定的分析包、Prompt、回填结果和报告

### 建议状态流转
- `created`：任务已创建
- `package_uploaded`：分析包已上传
- `prompt_generated`：Prompt 已生成
- `result_filled`：结果已回填
- `report_ready`：报告已生成
- `failed`：任务失败

### 设计目标
让整个工作流每一步都有可视状态，避免用户混乱。

---

## 6.3 分析包接收与解析模块

### 功能
接收本地预处理工具输出的标准分析包，并将其中内容解析入系统。

### 支持输入方式
- 上传 ZIP 包
- 上传目录中的核心文件（后续可选）

### 解析对象
- `task_manifest.json`
- `metadata.json`
- `segments.json`
- `transcript.txt`
- `contact_sheet.jpg`
- `frames/`
- `segment_keyframes/`

### 处理逻辑
1. 校验分析包完整性
2. 解压到任务目录
3. 读取 JSON 文件
4. 检查关键字段
5. 建立任务与分析包记录关联
6. 生成分析包摘要供页面展示

### 降级策略
若部分文件缺失，仍允许任务继续，只要最关键内容存在：
- `task_manifest.json`
- `metadata.json`
- 至少一张 `contact_sheet.jpg` 或关键帧图片

### 目标
让云端项目对“标准分析包”有稳定、可预期的读取能力。

---

## 6.4 Prompt 生成模块

### 功能
根据任务类型、分析包摘要、知识库内容和 Prompt 模板，自动生成供 ChatGPT / DeepSeek 使用的 Prompt。

### 输入来源
- 任务类型
- 视频元信息
- segments 摘要
- transcript 摘要
- Contact Sheet / 关键帧说明
- 知识库召回内容
- 预设 Prompt 模板

### 输出内容
- 标准 Prompt 正文
- 建议附带图片清单
- 建议追问问题
- 建议回填格式模板

### 首版要求
- 支持拆解模式 Prompt
- 支持优化模式 Prompt
- Prompt 尽量结构化，避免随意生成
- 支持复制按钮和导出文本

### 设计原则
Prompt 生成要尽量模板化和参数化，避免内容漂移太大。

---

## 6.5 知识库模块

### 功能
为 Prompt 生成和报告整理提供知识支撑。

### 首版定位
首版不做复杂 CMS，不做复杂向量数据库，优先保证内容可维护和可检索。

### 数据来源
- Markdown
- JSON
- YAML

### 管理方式
- 仓库内静态文件维护
- 系统启动时加载
- 支持后台页面浏览、搜索、查看

### 检索方式建议
首版使用轻量方案：
- 分类筛选
- 标签筛选
- 关键词搜索
- 简单相似规则

### 首版不建议
- 独立向量数据库
- 重型 embedding 服务
- 复杂召回排序链路

### 后续扩展
服务器升级后，可再引入轻量 embedding 检索。

---

## 6.6 回填结果接收模块

### 功能
接收用户从 ChatGPT / DeepSeek 获取的分析结果，并绑定到对应任务。

### 支持方式
- 直接粘贴文本
- 上传 Markdown 文件
- 上传纯文本文件

### 处理逻辑
1. 用户选择任务
2. 粘贴或上传分析结果
3. 记录来源模型
4. 保存原始文本
5. 进入结构化整理阶段

### 首版目标
- 原始文本必须完整保存
- 不强依赖高精度自动解析
- 允许后续人工补充标签和摘要

### 设计原则
先确保“能收进来”，再逐步优化结构化解析质量。

---

## 6.7 报告生成与展示模块

### 功能
把回填结果整理为项目内的标准报告页面。

### 报告类型
#### 拆解模式报告
- 总体结论
- 风格标签
- 调色推测
- 剪辑结构分析
- 片段拆解
- 疑似 AI 生成片段提示
- 仿拍方案

#### 优化模式报告
- 总体判断
- 必须改
- 建议改
- 可选优化
- 调色建议
- 剪辑建议
- 风格提升建议

### 首版策略
- 先支持“按模板结构保存和展示”
- 允许部分内容人工整理后保存
- 报告页支持 Markdown 风格阅读

### 导出能力
- 导出 Markdown
- 导出纯文本
- PDF 导出后续可选

---

## 6.8 历史记录模块

### 功能
保存并展示所有历史任务，便于用户回看、复用和沉淀经验。

### 能力要求
- 任务列表页
- 按任务类型筛选
- 按时间倒序显示
- 查看 Prompt 历史
- 查看回填原文
- 查看最终报告
- 删除任务
- 重新生成 Prompt

### 价值
这是智摄从“临时工具”走向“长期创作工作流系统”的关键模块。

---

## 6.9 模板与配置模块

### 功能
管理与 Prompt、输出结构、系统行为相关的配置。

### 建议管理对象
- Prompt 模板
- 拆解模式模板
- 优化模式模板
- 建议追问模板
- 回填模板
- 系统基础设置
- 历史保留策略

### 首版实现建议
- 模板使用文件存储
- 后台只做浏览与选择
- 编辑功能后续再加

---

## 6.10 存储与文件管理模块

### 功能
统一管理分析包、回填结果、报告和知识库文件。

### 设计原则
- 所有文件按任务目录归档
- 命名统一
- 路径可配置
- 易于备份

### 建议目录结构

```text
storage/
  tasks/
    task_0001/
      package/
      prompt/
      result/
      report/
  knowledge/
  exports/
```

### 说明
- `package/`：分析包解压后内容
- `prompt/`：生成的 Prompt 文本
- `result/`：原始回填文本
- `report/`：结构化报告
- `exports/`：导出文件

## 7. 数据流设计

## 7.1 新建任务数据流

```text
用户新建任务
  ↓
系统生成 task_id
  ↓
用户上传分析包
  ↓
系统校验、解压、解析
  ↓
保存任务记录与分析包摘要
  ↓
任务状态变为 package_uploaded
```

## 7.2 Prompt 生成数据流

```text
用户点击生成 Prompt
  ↓
系统读取任务类型 + 分析包摘要
  ↓
系统检索知识库
  ↓
系统套用 Prompt 模板
  ↓
生成标准 Prompt + 追问建议 + 回填模板
  ↓
任务状态变为 prompt_generated
```

## 7.3 结果回填数据流

```text
用户粘贴或上传 ChatGPT 结果
  ↓
系统保存原始文本
  ↓
系统生成结构化报告
  ↓
保存报告文件
  ↓
任务状态变为 report_ready
```

## 9. 数据库设计原则

考虑服务器只有 2 核 2 GB，首版建议使用 **SQLite**。

### 原则
1. 先满足单用户稳定运行
2. 数据结构简单清晰
3. 不引入重数据库服务
4. 大文件不进数据库，数据库只保存索引和元数据

### 首版建议核心表
- `tasks`
- `packages`
- `prompts`
- `results`
- `reports`
- `knowledge_index`
- `settings`

### 数据库存什么
- 任务基础信息
- 状态
- 分析包摘要
- Prompt 文本索引
- 回填记录索引
- 报告索引
- 文件路径
- 标签与元信息

### 数据库不存什么
- 大图片二进制
- ZIP 包二进制
- 原始视频
- 大型文档原文（如无必要）

---

## 10. 文件系统设计原则

由于首版文件主要是图片、JSON、文本和 Markdown，建议优先使用本地磁盘目录存储。

### 原则
- 数据库存索引
- 文件系统存实体文件
- 文件名和路径规则统一
- 任务删除时支持级联清理文件

### 好处
- 节省数据库资源
- 便于手动备份
- 与 Docker 卷挂载兼容性好

## 11. 部署架构建议

### 11.1 首版推荐部署方式

首版继续采用 **Docker Compose + 尽量少容器 + 统一公网网关** 的方式部署，并直接兼容现有服务器上线思路。

建议口径如下：

- 本地项目目录：`d:\apps\Photography`
- 服务器运行目录：`/opt/apps/photography`
- Docker 共享网络：`shared_gateway`
- 预览访问：`http://127.0.0.1:18120`
- 正式入口：`https://<PHOTOGRAPHY_DOMAIN>`
- 公网网关事实来源：`/opt/apps/hiremate/Caddyfile`

标准上线顺序建议：

1. 本地 Git 提交并推送到远端仓库。
2. 服务器在 `/opt/apps/photography` 执行 `git pull`。
3. 先启动智摄容器并通过本机端口做预览验证。
4. 验证通过后，再把智摄接入统一 Caddy 网关。
5. 如涉及存储结构或数据库升级，先备份再执行重建。

### 11.2 推荐容器组成

#### 容器 1：`photography-app`
负责：
- FastAPI 服务
- 页面渲染
- API 接口
- 任务与报告逻辑
- 文件处理与解析

说明：
- 如果首版采用 FastAPI + Jinja2，一台 `photography-app` 就可以同时承接页面与 API。
- 该容器建议接入 `shared_gateway`，但不直接绑定公网 `80/443`。

#### 容器 2：`photography-web`（可选）
适用场景：
- 需要独立处理静态资源缓存
- 下载文件响应头需要单独收口
- 后续页面层改成轻前端

首版如无明确需要，可先不拆这个容器。

### 11.3 服务器目录建议

```text
/opt/apps/photography/
  ├── .env
  ├── docker-compose.yml
  ├── README.md
  ├── server/
  ├── deploy/
  │   └── Caddyfile.photography.example
  └── volumes/
      ├── sqlite/
      ├── storage/
      └── logs/
```

说明：
- `volumes/sqlite/`：存放 SQLite 数据库文件
- `volumes/storage/`：存放分析包解压目录、导出文件和知识库文件
- `volumes/logs/`：存放应用日志，便于宿主机直接查看和备份
- `deploy/`：存放接入统一网关时需要引用的示例配置

### 11.4 统一网关接入建议

智摄首版建议采用“同域页面 + 同域 API”模式，不单独拆 API 公网域名。Caddy 参考片段可定义为：

```caddy
<PHOTOGRAPHY_DOMAIN> {
    import common_site

    @backend path /api/* /uploads/* /exports/*
    handle @backend {
        reverse_proxy photography-app:8000
    }

    handle {
        reverse_proxy photography-app:8000
    }
}
```

这意味着：
- 前台页面、登录态、API 与导出路径共用同一个正式域名
- 上传、下载和页面访问都经由统一公网入口转发
- 如域名、备案或 HTTPS 尚未准备完成，先停留在 `127.0.0.1` 预览阶段即可

### 11.5 首版不建议增加的容器
- Redis
- PostgreSQL
- Celery Worker
- MinIO
- 向量数据库

### 11.6 原因
在 2 核 2 GB 机器上，上述组件会显著提高内存占用和维护成本，而首版并不需要。

## 12. 技术选型建议

### 12.1 后端框架
建议：**FastAPI**

原因：
- 轻量
- 易写 API
- 易写文件上传逻辑
- 易配合 Jinja2 模板
- 适合 Python 生态，与本地工具风格统一

### 12.2 页面层
建议两种路线二选一：

#### 路线 A：FastAPI + Jinja2
优点：
- 最轻量
- 部署简单
- 适合单用户后台式系统

#### 路线 B：轻前端 + FastAPI API
优点：
- 页面更灵活
- 后续扩展方便

对于当前资源，**优先推荐路线 A**。

### 12.3 数据库
建议：**SQLite**

### 12.4 文件存储
建议：**宿主机挂载目录**

### 12.5 反向代理
建议：**统一复用现有 Caddy 网关**

智摄项目自身可以不再额外承担公网 HTTPS，只需要保证应用容器正确接入 `shared_gateway`。

## 15. 日志与监控策略

### 首版日志需求
记录：
- 登录事件
- 任务创建事件
- 分析包上传事件
- Prompt 生成事件
- 回填保存事件
- 报告生成事件
- 错误与异常

### 首版建议
- 使用文本日志
- 按日期切分
- 保留最近日志
- 应支持 `docker compose logs` 与宿主机日志目录双向排查
- 健康检查默认使用 `GET /api/health`

### 首版不做
- 复杂监控平台
- 分布式追踪
- 指标大盘

对于单用户轻服务，文本日志足够。

## 16. 错误处理策略

## 17. 安全设计原则

### 首版安全重点
- 基础登录保护
- 敏感配置不写死在前端
- 上传文件类型校验
- 路径遍历防护
- 文件大小限制
- 服务仅个人使用

### 首版不做
- 复杂权限系统
- 多因子认证
- 企业级审计体系

但至少要做到：
- 不裸奔
- 不随意暴露文件目录
- 不让任意上传文件直接执行

## 18. 性能设计原则

考虑 2 核 2 GB 服务器，首版性能目标应非常明确：

### 核心目标
- 单用户使用流畅
- 页面响应稳定
- 上传分析包可接受
- Prompt 生成迅速
- 回填保存快速

### 性能策略
- 不做重处理
- 上传包大小限制
- 大图按需展示
- 文件异步保存可后续考虑
- 避免大量同时操作

### 首版容量建议
- 每任务关键帧数量有限制
- 分析包建议控制在合理大小范围内
- 历史任务支持定期清理

## 21. 验收标准

若首版云端轻服务达到以下标准，则认为架构设计成功：

1. 能在 2 核 2 GB 云服务器上稳定运行。
2. 能正常接收并解析标准分析包。
3. 能稳定生成两类 Prompt（拆解 / 优化）。
4. 用户能顺畅完成“上传分析包 → 复制 Prompt → 回填结果 → 查看报告”的完整流程。
5. 历史任务可回看、可搜索、可删除。
6. 系统维护成本低，Git 更新和 Docker 重启简单。
