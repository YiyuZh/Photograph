# 数据库表结构草案

## 1. 文档定位

本文档用于定义 **智摄项目首版云端轻服务** 的数据库表结构草案。

当前数据库设计必须严格服务于以下前提：
- 单用户个人使用
- 云服务器配置为 2 核 2 GB
- 首版采用轻量架构
- 数据库存储索引与元数据，不存储大体积文件本体
- 原视频不作为云端核心存储对象
- 重点支持“分析包 → Prompt → 回填结果 → 报告归档”的工作流
- 生产部署目录将固定为 `/opt/apps/photography`
- SQLite 文件与存储目录需位于宿主机挂载卷，而不是镜像层

因此，数据库设计原则应为：

- **轻量**
- **清晰**
- **稳定**
- **可维护**
- **便于后续扩展**

首版推荐数据库：**SQLite**

---

## 2. 总体设计原则

### 2.1 数据库存什么

数据库主要存：
- 用户与登录配置
- 任务基础信息
- 分析包摘要信息
- Prompt 记录
- 回填结果记录
- 报告记录
- 知识库索引信息
- 模板与系统配置
- 标签与关联关系

### 2.2 数据库不存什么

数据库不存：
- 原始视频文件
- ZIP 包二进制本体
- 图片二进制本体
- Contact Sheet 二进制本体
- 音频文件二进制本体
- 大量原始 Markdown / 文本附件（如可放文件系统）

这些内容统一放在文件系统目录中，数据库只保存：
- 路径
- 文件名
- 摘要
- 关联关系

### 2.3 设计目标

1. 首版表结构尽量少而完整。
2. 允许部分字段为空，适配“部分成功”的预处理与回填流程。
3. 状态字段清晰，便于前端页面展示任务流转。
4. 后续若迁移到 PostgreSQL，表结构能平滑过渡。
5. 路径类字段优先保存相对于 `storage_root` 的相对路径，避免绑定某一台机器的绝对路径。

---

## 3. 核心实体总览

首版建议包含以下核心表：

1. `users`：用户表
2. `tasks`：任务主表
3. `packages`：分析包表
4. `package_files`：分析包文件索引表
5. `segments`：片段摘要表
6. `prompts`：Prompt 记录表
7. `prompt_templates`：Prompt 模板表
8. `results`：回填结果表
9. `reports`：报告表
10. `knowledge_items`：知识库条目表
11. `knowledge_tags`：知识库标签表
12. `task_tags`：任务标签表
13. `task_tag_relations`：任务标签关联表
14. `settings`：系统设置表
15. `audit_logs`：关键操作日志表（可选但推荐）

如果首版想再进一步收敛，也可以暂时不做：
- `knowledge_tags`
- `task_tags`
- `task_tag_relations`
- `audit_logs`

但从中期维护角度，建议保留设计。

## 4. 表结构详细设计

## 4.1 users

### 用途
用于单用户登录与基础账户信息管理。

### 建议字段

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | INTEGER | 主键，自增 |
| username | TEXT | 用户名，唯一 |
| password_hash | TEXT | 密码哈希 |
| display_name | TEXT | 显示名称，可空 |
| is_active | INTEGER | 是否启用，0/1 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |
| last_login_at | TEXT | 最近登录时间，可空 |

### 约束建议
- `username` 唯一
- 首版仅一条有效用户记录即可

### 备注
虽然是单用户项目，但保留用户表更利于后续扩展，也利于登录逻辑清晰化。

## 8. 字段类型建议

由于首版使用 SQLite，建议采用以下约定：

- 主键：`INTEGER PRIMARY KEY AUTOINCREMENT`
- 时间：统一 `TEXT`，采用 ISO 8601 格式
- 布尔：`INTEGER`，0/1
- JSON：首版先用 `TEXT` 存字符串化 JSON
- 路径：`TEXT`
- 大段内容：`TEXT`

### 时间格式建议
统一使用：
- `2026-04-18T12:34:56+08:00`

这样后续迁移数据库更平滑。

### 路径字段建议
- 所有 `*_path` 字段建议保存相对于 `storage_root` 的相对路径
- `storage_root` 由环境变量或 `settings` 表统一配置
- 这样可以兼容本地 Windows 路径与生产机 Linux 路径差异

---

## 9. 删除策略建议

首版建议采用 **软删除 + 定期清理文件** 的方式。

### tasks
- 增加 `deleted_at`
- 页面默认不展示已删除任务

### 物理文件
- 删除任务时可先标记
- 后续清理脚本定期删除对应目录

### 好处
- 降低误删风险
- 便于回滚和排错

---

## 10. 版本化策略建议

以下对象建议支持“多版本但只有一个当前版本”：

### prompts
- `version`
- `is_current`

### reports
- `version`
- `is_current`

### 模板
- `version`
- `is_default`

### 好处
- 用户重复生成 Prompt 时不会覆盖历史
- 报告可以重新整理后保留旧版本

## 11. 首版最小 ER 思路

可以把首版数据库理解成下面这张简化关系图：

```text
users (1)
  └── (N) tasks
         ├── (1) packages
         │      ├── (N) package_files
         │      └── (N) segments
         ├── (N) prompts
         ├── (N) results
         └── (N) reports

knowledge_items
prompt_templates
settings
```

如果首版要继续收敛，最小闭环甚至可以只先做：
- `users`
- `tasks`
- `packages`
- `prompts`
- `results`
- `reports`
- `knowledge_items`
- `settings`

但从实际页面展示角度，`package_files` 和 `segments` 还是建议首版就保留。

- 默认拆解模式模板

## 13. 迁移与演进建议

### 首版阶段
- 直接使用 SQLite
- 用 SQLAlchemy / SQLModel / Peewee 等 ORM 管理表
- 通过迁移工具管理 schema 变更
- 数据库文件建议固定在 `/opt/apps/photography/volumes/sqlite/photography.db`
- 发布前把 SQLite 文件备份到 `/opt/backups/releases/<release_id>/photography/sqlite/`

### 第二阶段
如果未来出现以下情况，可考虑迁移 PostgreSQL：
- 多用户使用
- 数据量明显上升
- 需要更复杂查询
- 需要更高并发稳定性

### 迁移注意事项
当前设计已尽量避免使用 SQLite 专属奇技淫巧，因此迁移成本不会太高。

补充建议：
- 不要把 SQLite 文件放在容器镜像层或临时目录
- 不要在升级时使用会删除卷数据的命令
- 与宿主机文件系统强相关的路径，应通过 `storage_root` 统一换算

## 14. 首版建表顺序建议

建议按以下顺序建表：

1. `users`
2. `tasks`
3. `packages`
4. `package_files`
5. `segments`
6. `prompt_templates`
7. `prompts`
8. `results`
9. `reports`
10. `knowledge_items`
11. `settings`
12. 其余标签与日志表

这个顺序更符合开发时从主流程往外扩展的思路。

---

## 15. 验收标准

如果数据库草案满足以下要求，则认为可进入下一阶段：

1. 能完整覆盖“分析包 → Prompt → 回填 → 报告”的流程。
2. 表数量不失控，首版维护成本可接受。
3. 文件本体与数据库索引分离清晰。
4. 关键对象支持版本化与状态管理。
5. 后续页面和 API 基本都能从这些表中直接找到数据来源。
6. 后续迁移 PostgreSQL 仍有空间。

## 16. 一句话总结

**智摄首版数据库的核心任务，不是存一切，而是把任务、分析包、Prompt、结果和报告之间的关系存清楚。**
