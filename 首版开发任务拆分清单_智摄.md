# 首版开发任务拆分清单

## 1. 文档定位

本文档用于将 **智摄首版** 的开发工作拆分为一组可执行、可排期、可验收的任务。

它服务于以下目标：
- 让 Codex 或开发者明确先做什么、后做什么
- 避免一开始同时铺太多模块导致失控
- 确保首版始终围绕最小可用闭环推进
- 适配当前明确约束：**单用户、2 核 2 GB、Docker 部署、云端轻服务、本地预处理、手动 AI 深分析**

---

## 2. 首版总目标

首版必须打通的最小闭环是：

**创建任务 → 上传分析包 → 解析分析包 → 生成 Prompt → 回填 AI 结果 → 生成报告 → 查看历史记录**

所有任务拆分都必须围绕这条链路服务。

---

## 3. 开发拆分原则

### 3.1 原则

1. **先骨架，后功能，最后补体验**
2. **先打通主流程，再补边缘功能**
3. **先实现稳定的最小能力，再考虑优化和美化**
4. **每个阶段结束都应能运行、能验证、能演示**
5. **不要提前做未来资源条件下才需要的能力**

### 3.2 开发顺序要求

首版必须按以下顺序推进：

1. 项目骨架与环境初始化
2. 数据库与模型
3. 登录与基础页面
4. 任务管理
5. 分析包上传与解析
6. Prompt 生成
7. 结果回填
8. 报告生成与展示
9. 知识库与设置
10. Docker、README、测试夹具与收尾

---

## 4. 阶段总览

建议将首版拆成 8 个阶段：

### 阶段 A：项目初始化与工程骨架
### 阶段 B：数据库与基础模型
### 阶段 C：认证、页面骨架与任务管理
### 阶段 D：分析包上传与解析
### 阶段 E：Prompt 模块
### 阶段 F：结果回填与报告模块
### 阶段 G：知识库、设置与历史页补全
### 阶段 H：部署、测试、文档与交付收尾

## 13. 任务依赖关系

为了避免乱序开发，依赖关系如下：

### A 阶段完成后，才能做 B
### B 阶段完成后，才能稳定推进 C
### C 阶段完成后，才能做 D
### D 阶段完成后，才能做 E
### E 阶段完成后，才能做 F
### F 阶段完成后，再补 G
### H 阶段应在主链路完成后集中收尾

简化依赖关系：

```text
A 项目骨架
  ↓
B 数据库模型
  ↓
C 登录 + 任务
  ↓
D 分析包上传与解析
  ↓
E Prompt 生成
  ↓
F 回填与报告
  ↓
G 知识库 + 设置 + 首页补全
  ↓
H Docker + README + 测试夹具 + 自测
```

1. 不要跨阶段乱实现功能
2. 每一阶段都要保证代码可运行
3. 每次提交都要说明完成了哪些任务编号
4. 遇到文档冲突时，优先保证主链路正确和低资源适配
5. 不要在首版引入不必要的新技术栈

---

## 17. 最终验收口径

首版只有在以下条件全部满足时，才算真正完成：

1. 用户可以登录系统
2. 用户可以创建拆解或优化任务
3. 用户可以上传分析包并看到解析摘要
4. 用户可以生成 Prompt 并复制到 ChatGPT
5. 用户可以把 AI 分析结果回填系统
6. 用户可以生成并查看报告
7. 用户可以在历史记录中再次查看任务与报告
8. 整个系统可以在 Docker 环境中稳定启动
9. 整体架构没有违背 2 核 2 GB 服务器约束

---

## 18. 一句话总结

**首版开发任务拆分的核心，不是把功能列满，而是确保智摄能以最小成本先跑通“分析包 → Prompt → 回填 → 报告”这条真正有价值的主链路。**
