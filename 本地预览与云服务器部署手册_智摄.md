# 本地预览与云服务器部署手册_智摄

这份手册是给 **智摄首版云端轻服务** 用的正式部署说明，目标是让你只看这一份文档，就能完成：

1. 本地把项目跑起来
2. 把本地目录接入 Git
3. 推送到私有仓库
4. 在云服务器上配置 deploy key
5. 首次部署到 `/opt/apps/photography`
6. 先用 `127.0.0.1:18120` 做本机预览
7. 验证通过后再接入统一 Caddy 网关
8. 后续更新、回滚、排查问题

这次手册继续沿用智摄现有的项目结构和上线思路，不推翻重来。

## 1. 文档适用前提

### 1.1 这份手册适合什么场景

- 你正在维护 `D:\apps\Photography` 这套智摄项目目录

- 本地源码目录：`D:\apps\Photography\server`
- 服务器项目根目录：`/opt/apps/photography`

## 3. 本地开发、Git 与 Docker 预览

### 3.1 先确认你的本地目录现状

当前你这台机器上的 `D:\apps\Photography` 之前并不是一个 Git 仓库。

这意味着：

- 你不能直接 `git pull`
- 你也不能指望服务器去 clone 一个不存在的远端仓库
- 正确顺序应该是：先把本地目录接入 Git，再推到代码托管平台，然后服务器再去拉

### 3.2 第一次把本地目录变成 Git 仓库

在 **`D:\apps\Photography` 根目录** 执行：

```powershell
cd D:\apps\Photography
git init
git branch -M main
git status
```

预期结果：

- `git status` 能正常输出当前文件状态
- 当前目录已经被 Git 接管

失败时优先检查：

- 本机是否安装了 Git
- 当前目录是否输错成了 `D:\apps\Photography\server`

### 3.3 在代码托管平台创建私有仓库

你需要先在 GitHub 或其他 Git 平台手动创建一个 **私有仓库**。

推荐做法：

- 你当前实际使用的 GitHub 仓库名是 `Photograph`
- 分支主线使用 `main`
- 创建空仓库，不要自动加 README、.gitignore、license

创建完成后，你会拿到 SSH 地址，例如：

```text
git@github.com:YiyuZh/Photograph.git
```

这里的 `<GITHUB_REPO_SSH>` 不是你的“个人 SSH 密钥”，也不是后面服务器生成的 `deploy key`。

它指的是：**这个 GitHub 仓库本身的 SSH 克隆地址**。  
你真正要替换成的内容，通常长这样：

```text
git@github.com:YiyuZh/Photograph.git
```

你可以在 GitHub 仓库页面点：

1. `Code`
2. 切到 `SSH`
3. 复制那一行地址

这一行地址只会用在：

- `git remote add origin <GITHUB_REPO_SSH>`
- 你本地电脑上的 Git 推送与仓库绑定

如果服务器是单项目环境，也可以直接拿它做 `git clone`。  
但你当前这台服务器是多项目环境，服务器侧会在后面的 `4.5` 到 `4.7` 里改成 **带 alias 的 SSH 地址**，避免不同项目的 deploy key 互相干扰。

你可以先把它理解成：**仓库地址**，不是**密钥内容**。

### 3.4 第一次提交并推送本地代码

继续在 **`D:\apps\Photography` 根目录** 执行：

```powershell

```

预期结果：

- 远端仓库里出现 `main` 分支
- 如打 tag，则远端也能看到 `photography-v1.0.0`

失败时优先检查：

- 你本机是否已经配置了自己的 GitHub SSH key
- `git remote -v` 显示的地址是否正确
- 远端仓库是否真的是空仓库

## 4. 服务器首次部署

### 4.1 服务器前置检查

先登录服务器，执行：

```bash
uname -a
git --version
docker -v

```

预期结果：

- 服务器存在 Git
- 服务器存在 Docker 和 Docker Compose

失败时优先检查：

- 是否已经安装基础运行环境
- 当前用户是否具备 Docker 使用权限

### 4.2 服务器创建发布目录和备份目录

```bash
export SERVER_PUBLIC_IP=<SERVER_PUBLIC_IP>
export PHOTOGRAPHY_DOMAIN=photo.zenithy.art
export RELEASE_ID=$(date +%F-%H%M%S)

mkdir -p /opt/apps
mkdir -p /opt/apps/backups/releases/$RELEASE_ID
```

预期结果：

- `/opt/apps`
- `/opt/apps/backups/releases/$RELEASE_ID`

都已经存在。

### 4.3 在服务器上创建智摄专用 deploy key

这一步必须在 **服务器** 上执行，不是在你本地电脑上。

这里开始出现第二种和 SSH 有关的东西，它和上面的 `<GITHUB_REPO_SSH>` 不是一回事：

1. `<GITHUB_REPO_SSH>`：仓库地址，用来告诉 Git “去哪个仓库拉代码”
2. `deploy key`：服务器上生成的一对密钥，用来证明“这台服务器有权限拉这个私有仓库”

如果你之前的做法是：

- 先 SSH 登录服务器
- 在服务器里运行 `ssh-keygen`
- 把生成出来的 `.pub` 公钥内容复制到 GitHub 仓库的 `Deploy keys`

那你记忆里的流程就是对的，下面这几步就是在做这件事。

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t ed25519 -C "photography-prod" -f ~/.ssh/id_ed25519_photography -N ""
chmod 600 ~/.ssh/id_ed25519_photography
chmod 644 ~/.ssh/id_ed25519_photography.pub
cat ~/.ssh/id_ed25519_photography.pub
```

把输出的公钥完整复制出来。

预期结果：

- 生成两个文件：
  - `~/.ssh/id_ed25519_photography`
  - `~/.ssh/id_ed25519_photography.pub`

失败时优先检查：

- `~/.ssh` 权限是否正确
- 是否误在本机执行了这一步

### 4.4 把 deploy key 加到远端私有仓库

如果你用的是 GitHub：

1. 打开智摄私有仓库
2. 进入 `Settings`
3. 打开 `Deploy keys`
4. 点击 `Add deploy key`
5. 标题填：`photography-prod`
6. 内容粘贴 `id_ed25519_photography.pub`
7. **不要勾选 write access**

这里建议只给只读权限，因为服务器只需要拉代码，不需要推代码。

到这里你粘贴到 GitHub 的，不是 `<GITHUB_REPO_SSH>`，而是下面这个命令输出出来的**公钥内容**：

```bash
cat ~/.ssh/id_ed25519_photography.pub
```

### 4.5 让服务器拉代码时使用这把 key

这一步要根据服务器上是不是只有这一个 GitHub 私有仓库来决定。

你这台服务器已经不止一个项目，而且多个项目都会通过 Git 拉代码，所以 **不推荐** 直接把智摄这把 key 绑定到通用的 `Host github.com`。

原因是：

- `Host github.com` 是全局匹配
- 你一旦把它直接指向 `id_ed25519_photography`，这台服务器上别的 GitHub 私有仓库也可能被迫走这把 key
- 如果别的仓库绑定的是别的 deploy key，就容易出现“这个项目能拉，另一个项目突然拉不下来”

### 推荐做法：给智摄单独配置一个 Host alias

```bash
cp ~/.ssh/config ~/.ssh/config.bak.$(date +%F-%H%M%S) 2>/dev/null || true

cat >> ~/.ssh/config <<'EOF'
Host github.com-photography
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_photography
    IdentitiesOnly yes
EOF

chmod 600 ~/.ssh/config
ssh -T git@github.com-photography
```

预期结果：

- SSH 测试成功
- GitHub 返回认证成功提示
- 常见成功返回类似：

```text
Hi <你的 GitHub 用户名>/<仓库名>! You've successfully authenticated, but GitHub does not provide shell access.
```

- 这句话不是报错，它表示：
  - 服务器已经成功使用智摄这把 deploy key 连接上 GitHub
  - GitHub 认可这把 key 对目标仓库有读取权限
  - “does not provide shell access” 是 GitHub 的正常提示，意思只是 GitHub 不提供交互式 shell

失败时优先检查：

- deploy key 是否已经加到正确仓库
- 你复制的是否是 `.pub` 公钥而不是私钥
- `~/.ssh/config` 是否写错缩进或文件权限
- 是否误把别的项目已有配置覆盖掉了

注意：

- 这里用的是 `github.com-photography`，不是默认的 `github.com`
- 这样做的目的是让智摄只使用自己的 key，不影响服务器上的其他项目
- 如果服务器上还有别的私有仓库，也建议每个仓库各自使用一把 deploy key 和一个独立 alias
- 如果 `~/.ssh/config` 已经存在，请追加配置，不要整文件覆盖

### 如果服务器未来只拉这一套仓库

只有在这台服务器确实只需要拉智摄这一套 GitHub 私有仓库时，才可以用下面这种更简化的全局写法：

```bash
cat > ~/.ssh/config <<'EOF'
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_photography
    IdentitiesOnly yes
EOF

chmod 600 ~/.ssh/config
ssh -T git@github.com
```

但按你现在这台服务器的实际情况，不建议使用这一种。

### 4.6 验证服务器能读取远端仓库

如果你使用的是上面的 `github.com-photography` alias，那么服务器侧这里不要再直接写 GitHub 默认 SSH 地址，而要写成 alias 版本的仓库地址。

你当前这套智摄项目，服务器侧实际应写成：

失败时优先检查：

- alias 形式的仓库地址是否正确

### 4.7 第一次 clone 或已有目录 pull

如果服务器上还没有智摄目录：

```bash
git clone git@github.com-photography:YiyuZh/Photograph.git /opt/apps/photography
```

如果服务器上已经有目录：

预期结果：

### 4.8 创建服务器 `.env`

```bash
cd /opt/apps/photography/server
cp .env.example .env
export SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_urlsafe(48))")
export DEFAULT_PASSWORD=$(python3 -c "import secrets,string; chars=string.ascii_letters + string.digits + '!@#$%^&*()-_=+'; print(''.join(secrets.choice(chars) for _ in range(20)))")
printf 'SECRET_KEY=%s\nDEFAULT_PASSWORD=%s\n' "$SECRET_KEY" "$DEFAULT_PASSWORD"
nano .env
```

这三个字段必须改：

- `SECRET_KEY`
- `DEFAULT_PASSWORD`
- `APP_BASE_URL`

推荐至少改成：

```env
SECRET_KEY=<填上面服务器刚生成的随机字符串>
DEFAULT_PASSWORD=<填上面服务器刚生成的强密码>
APP_BASE_URL=https://photo.zenithy.art
```

上面两条命令已经会在服务器上生成：

- 一条高强度随机 `SECRET_KEY`
- 一条可直接登录后台的初始强密码 `DEFAULT_PASSWORD`

你只需要把终端打印出来的这两个值复制进 `.env` 即可。

如果你当前只想做服务器本机预览，可以临时写：

```env
APP_BASE_URL=http://127.0.0.1:18120
```

但最终公网切流前，必须改回：

```env
APP_BASE_URL=https://photo.zenithy.art
```

这里的 `APP_BASE_URL` 不是随便写一个你喜欢的网站地址就行。

它应该填写为：**这个智摄项目最终正式对外访问的域名地址**。

也就是说：

- 如果你最后准备让智摄挂在 `https://photo.yourdomain.com`
- 并且 DNS 解析和 Caddy 反代最后也会接到这个域名

那这里就应该写：

```env
APP_BASE_URL=https://photo.yourdomain.com
```

它的用途不是“装饰用域名”，而是项目在生成报告下载链接、公开访问地址等场景时，要知道“自己最终对外是什么地址”。

所以规则是：

- 只做服务器本机预览时：可以临时写 `http://127.0.0.1:18120`
- 准备正式上线时：改成你最终实际要解析到这台服务器的正式域名
- 不要写成和最终解析、Caddy 配置不一致的域名

## 5. 统一网关接入

### 5.1 先明确网关事实来源

智摄项目自身不维护正式公网网关配置。  
真正要改的文件始终是：

- `/opt/apps/hiremate/Caddyfile`

### 5.2 改前先备份

```bash
cp /opt/apps/hiremate/Caddyfile /opt/apps/backups/releases/$RELEASE_ID/Caddyfile.photography.bak
```

### 5.3 把智摄站点块加到 Caddyfile

把下面这段加入 `/opt/apps/hiremate/Caddyfile`：

```caddy
photo.zenithy.art {
    import common_site

    @backend path /api/* /uploads/* /exports/*
    handle @backend {
        reverse_proxy photography-app:8000
    }

    handle {
        reverse_proxy photography-app:8000
    }
}
```

说明：

- 页面和 API 共用同一个域名
- 智摄仍然不直接绑定宿主机 `80/443`
- 统一公网流量继续交给现有 Caddy

### 5.4 格式化、校验、重载

```bash
caddy fmt --overwrite /opt/apps/hiremate/Caddyfile
caddy validate --config /opt/apps/hiremate/Caddyfile
caddy reload --config /opt/apps/hiremate/Caddyfile
```

预期结果：

- `validate` 通过
- `reload` 成功

失败时优先检查：

- Caddyfile 是否漏花括号
- `photography-app:8000` 在现有网关环境里是否可达

## 7. 日常更新与回滚

### 7.1 日常更新标准流程

本地改完后：

```powershell
cd D:\apps\Photography
git status
git add .
git commit -m "feat: describe your change"
git push
```

服务器拉最新：

```bash
cd /opt/apps/photography
git pull
cd /opt/apps/photography/server
docker compose up -d --build
docker compose ps
curl -fsS http://127.0.0.1:18120/api/health
```

如果服务器部署固定 tag：

```bash
cd /opt/apps/photography
git fetch --tags
git checkout <你要部署的 tag>
cd /opt/apps/photography/server
docker compose up -d --build
```

### 7.2 更新前备份

```bash
cp /opt/apps/photography/server/.env /opt/apps/backups/releases/$RELEASE_ID/.env.bak
cp -a /opt/apps/photography/server/storage /opt/apps/backups/releases/$RELEASE_ID/storage.bak
cp -a /opt/apps/photography/server/logs /opt/apps/backups/releases/$RELEASE_ID/logs.bak
cp /opt/apps/hiremate/Caddyfile /opt/apps/backups/releases/$RELEASE_ID/Caddyfile.bak
```

### 7.3 回滚原则

- 禁止使用 `docker compose down -v`
- 禁止直接删 `storage/sqlite/photography.db`
- 代码回滚和数据备份要分开处理

### 7.4 代码回滚

回到上一个稳定 commit 或 tag：

```bash
cd /opt/apps/photography
git log --oneline -10
git checkout <上一稳定 commit 或 tag>
cd /opt/apps/photography/server
docker compose up -d --build
docker compose ps
curl -fsS http://127.0.0.1:18120/api/health
```

### 7.5 如果这次还改了 Caddy

先恢复旧的 Caddyfile，再 reload：

```bash
cp /opt/apps/backups/releases/$RELEASE_ID/Caddyfile.bak /opt/apps/hiremate/Caddyfile
caddy fmt --overwrite /opt/apps/hiremate/Caddyfile
caddy validate --config /opt/apps/hiremate/Caddyfile
caddy reload --config /opt/apps/hiremate/Caddyfile
```

## 8. 常见问题与排查

### 8.1 `git push` 失败

优先检查：

- 本地是否已经 `git remote add origin <GITHUB_REPO_SSH>`
- 本机是否配置了自己的 GitHub SSH key
- 远端私有仓库是否真的存在

### 8.2 服务器 `git clone` / `git pull` 失败

优先检查：

- deploy key 是否加到了正确仓库
- `ssh -T git@github.com-photography` 是否成功
- `git ls-remote git@github.com-photography:YiyuZh/Photograph.git` 是否成功
- 是否误把智摄 key 写到了通用 `Host github.com`，影响了其他项目

### 8.3 `docker` 或 `docker compose` 不存在

优先检查：

- 服务器是否已安装 Docker
- 当前用户是否有 Docker 权限

### 8.4 `127.0.0.1:18120` 无法访问

优先检查：

- `docker compose ps`
- `docker compose logs --tail=100 app`
- `18120` 是否被其他程序占用
- `.env` 是否正确

### 8.5 登录失败

优先检查：

- 你输入的是不是 `.env` 里的 `DEFAULT_USERNAME` / `DEFAULT_PASSWORD`
- 是否改过 `.env` 但没有重建容器

### 8.6 上传 ZIP 被拒绝

优先检查：

- 文件是否真的是 `.zip`
- 大小是否超过 `MAX_PACKAGE_UPLOAD_MB`
- ZIP 是否缺少核心文件：`task_manifest.json`、`metadata.json`、联系表图或关键帧

### 8.7 报告无法生成

优先检查：

- 是否已经保存至少一份结果回填
- 当前任务是否已经生成 Prompt

### 8.8 域名已解析但公网打不开

优先检查：

- DNS 是否已指向 `<SERVER_PUBLIC_IP>`
- `/opt/apps/hiremate/Caddyfile` 是否已经加上智摄站点块
- `caddy validate` / `caddy reload` 是否成功
- 现有统一网关环境是否真的能访问 `photography-app:8000`

## 一句话收口

智摄的正确上线顺序不是“改完直接暴露公网”，而是：

**本地改完 -> Git 提交推远端 -> 服务器 deploy key 拉代码 -> Docker 本机预览 -> 业务闭环验证 -> 最后接入统一 Caddy 网关。**
