## 01 任务目标

这个项目要解决的问题，是把智摄做成一套适合个人使用、轻量云服务器部署的 **视频 AI 分析工作流系统**。

它的核心目标不是做重型视频云处理，而是打通这条最小可用链路：

1. 本地预处理视频并生成分析包
2. 云端接收分析包并整理上下文
3. 生成高质量 Prompt
4. 用户手动把 Prompt 和素材提交给 ChatGPT / DeepSeek
5. 把模型输出回填到系统
6. 生成结构化报告并沉淀历史知识

同时，这个项目还要满足：

- 能在 `2 核 CPU + 2 GB RAM` 的云服务器稳定运行
- 使用 `FastAPI + Jinja2 + SQLite + Docker Compose`
- 保留原有 PRD 的轻量架构思路
- 接入现有统一网关体系，而不是单独造一套公网入口

## 03 已做决策 + 理由

- 采用 `FastAPI + Jinja2` 而不是前后端分离。
  理由：首版目标是轻量、低成本、快落地，服务器资源不足以支撑更重的前后端拆分成本。

- 采用 `SQLite` 而不是 `PostgreSQL`。
  理由：首版单用户、低并发、轻量部署，SQLite 更符合项目资源约束。

- 采用宿主机挂载目录保存 `storage/` 与 `logs/`。
  理由：这比对象存储或额外中间件更简单，更适合首版。

- 明确“智摄不直接占用公网 80/443”。
  理由：项目需要复用现有统一 Caddy 网关，避免与既有站点冲突。

- 部署路径统一为 `/opt/apps/photography/server`。
  理由：延续现有服务器上的 `/opt/apps/...` 部署思路，便于和其他项目统一管理。

- 本机预览端口固定为 `127.0.0.1:18120`。
  理由：先本机验证再公网切流，可以降低上线风险。

- 正式公网网关事实来源固定为 `/opt/apps/hiremate/Caddyfile`。
  理由：这是真实生效的公网入口来源，项目文档不能再把示例当正式配置。

- Prompt、结果、报告都支持历史版本和切换当前版本。
  理由：这是智摄工作流的重要特征，不能只保留单版本覆盖。

- 前端 UI 不直接照搬 AI 设计稿 HTML。
  理由：项目真实运行环境是 Jinja 模板，需要把视觉语言抽成公共样式和可复用结构，而不是导入一堆静态页面。

- 任务记录文档需要持续维护。
  理由：当前项目横跨 PRD、开发、部署、UI 和运维，如果没有持续记录，后续很容易丢失上下文和踩回头坑。

## 会话记录 2026-04-19 - 澄清部署手册中的仓库 SSH 地址与 deploy key 区别

### 本次任务要解决什么

用户阅读智摄部署手册时，对 `<GITHUB_REPO_SSH>` 和服务器生成的 `deploy key` 产生混淆，不确定手册里那一行 SSH 到底应该替换成什么，也不确定自己以前“在服务器生成密钥再粘贴到 GitHub”的做法是否仍然正确。

### 已做内容

- 重新核对部署手册中 Git、仓库地址、deploy key 的相关章节
- 在手册中明确补充了两类 SSH 的区别：
  - 仓库 SSH 地址：用于 `git remote add origin` 和 `git clone`
  - deploy key：在服务器生成，公钥粘贴到 GitHub 仓库 `Deploy keys`
- 在文档里补了一个明确例子：`git@github.com:<你的 GitHub 用户名或组织名>/Photography.git`
- 在 deploy key 章节补充说明：真正粘贴到 GitHub 的是 `cat ~/.ssh/id_ed25519_photography.pub` 输出的公钥内容

### 决策与理由

- 不改部署主流程，只增强文档中的“概念拆分”
- 理由是这次问题不是流程本身错了，而是“仓库地址”和“服务器公钥”都和 SSH 有关，初看很容易混为一体

### 错误与修复

- 错误：手册虽然写了 `git remote add origin <GITHUB_REPO_SSH>` 和 `ssh-keygen`，但没有足够直白地区分两者
